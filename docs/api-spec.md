## 1) Purpose and Scope

This document defines the production API contract for the Kluje enterprise platform. It translates the enterprise build specification into implementable endpoint standards, security requirements, request/response models, and operational constraints.

### In Scope
- REST API design under `/api/v1`
- Authentication, authorization, and tenant isolation
- Core domain endpoints (Auth, Leads, Routing, Providers, Billing, Ads, Reporting)
- Idempotency, pagination, filtering, and error models
- Webhook handling (Stripe + partner callbacks)
- Observability and reliability requirements for API operations

### Out of Scope
- Frontend UI behavior and component-level contracts
- Internal workflow implementation details (covered in `/docs/n8n-workflows.md`)
- Physical database DDL (covered in `/docs/db-schema.md`)

## 2) API Design Principles

1. **Tenant-safe by default:** Every tenant-bound endpoint must enforce organization-scoped access.
2. **Contract-first delivery:** OpenAPI 3.1 is the source of truth for schemas and generated clients.
3. **Idempotent mutations:** Routing and billing mutations require idempotency keys.
4. **Operational transparency:** Correlation IDs, structured errors, and audit event linkage for privileged actions.
5. **Backward-compatible evolution:** Non-breaking changes within `v1`; breaking changes require new API version.

---

## 3) Base URL, Versioning, and Content Negotiation

- **Base Path:** `/api/v1`
- **Protocol:** HTTPS only (TLS 1.2+ minimum)
- **Content Type:** `application/json`
- **Versioning:** URL-based (`/v1`) + schema `x-api-version` extension in OpenAPI
- **Character Encoding:** UTF-8

### Required Headers
- `Authorization: Bearer <jwt>` (except public auth bootstrap endpoints)
- `X-Tenant-Id: <uuid>` (required for tenant-scoped resources)
- `X-Correlation-Id: <uuid>` (optional from client; generated by API if absent)
- `Idempotency-Key: <opaque-string>` (required for selected POST endpoints)

### Response Headers
- `X-Request-Id`
- `X-Correlation-Id`
- `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`

## 4) Authentication and Authorization

## 4.1 Authentication Model

- JWT access tokens with short TTL (recommended: 15 minutes)
- Refresh tokens with rotation and reuse-detection
- Optional MFA challenge for admin roles and enterprise policy enforcement
- SSO/SAML enablement for enterprise plan tenants

### Auth Endpoints
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `POST /api/v1/auth/mfa/challenge`
- `POST /api/v1/auth/mfa/verify`
- `GET /api/v1/auth/me`

## 4.2 Authorization Model

Authorization layers:
1. **Identity Validation:** token signature, expiration, issuer/audience checks.
2. **Tenant Membership Check:** user must belong to `X-Tenant-Id`.
3. **Role Permission Check:** RBAC role grants required action.
4. **Resource Ownership/Scope Check:** enforce row-level access constraints.

### Suggested Role Matrix
- `tenant_admin`: full tenant configuration and billing access
- `ops_manager`: lead/routing/provider operations
- `finance_admin`: billing and invoices
- `analyst`: read-only reporting
- `provider_user`: provider profile and dispatch outcomes (scoped)

## 5) Common Request/Response Conventions

## 5.1 Pagination

- Cursor-based pagination for high-volume lists
- Query params:
  - `limit` (default 25, max 100)
  - `cursor` (opaque)
- Response envelope:

```json
{
  "data": [],
  "page": {
    "next_cursor": "opaque",
    "has_more": true
  }
}
```

## 5.2 Filtering and Sorting

- Standard query keys: `status`, `created_from`, `created_to`, `updated_from`, `updated_to`
- Search fields endpoint-specific (e.g., service category, geo, plan)
- Sort syntax: `sort=created_at:desc`

## 5.3 Error Envelope

```json
{
  "error": {
    "code": "ROUTING_CONSTRAINT_FAILED",
    "message": "No providers matched hard constraints",
    "details": {
      "service_category": "plumbing",
      "postal_code": "10001"
    },
    "request_id": "uuid",
    "correlation_id": "uuid"
  }
}
```

### Error Categories
- `400` validation errors
- `401` authentication failures
- `403` authorization denials
- `404` resource not found in tenant scope
- `409` idempotency or state conflict
- `422` domain rule violations
- `429` rate limit exceeded
- `500/502/503` internal or upstream failures

## 5.4 Idempotency Rules

Idempotency required for:
- `POST /api/v1/routing/evaluate`
- `POST /api/v1/routing/dispatch`
- `POST /api/v1/billing/subscriptions`
- `POST /api/v1/billing/webhooks/stripe` (event-id de-duplication)

Behavior:
- Same key + same payload returns original result
- Same key + different payload returns `409 IDEMPOTENCY_KEY_REUSE_MISMATCH`
- Key retention window recommended: 24–72 hours

## 6) Core Endpoint Domains

## 6.1 Auth

### `POST /api/v1/auth/login`
Authenticate user and issue token pair.

**Request**
```json
{
  "email": "user@tenant.com",
  "password": "********"
}
```

**Response**
```json
{
  "access_token": "jwt",
  "refresh_token": "opaque",
  "expires_in": 900,
  "mfa_required": false,
  "user": {
    "id": "uuid",
    "email": "user@tenant.com",
    "default_tenant_id": "uuid"
  }
}
```

## 6.2 Leads

### `POST /api/v1/leads`
Create normalized lead record.

Key rules:
- Validate required service and location fields
- Apply tenant policy checks (region/category permissions)
- Record append-only `lead_events` entry

### `GET /api/v1/leads/{leadId}`
Return lead, qualification status, and routing summary.

### `GET /api/v1/leads`
List leads with filters: `status`, `service_category`, date ranges.

## 6.3 Routing

### `POST /api/v1/routing/evaluate`
Run candidate selection + scoring in evaluation mode.

**Modes**
- `commit=false` dry-run simulation
- `commit=true` persisted decision

### `POST /api/v1/routing/dispatch`
Dispatch selected lead to top-N providers with throttling.

Dispatch controls:
- `max_targets`
- `channel_preferences` (API/email/SMS)
- `retry_policy` override (bounded)

## 6.4 Providers

### `GET /api/v1/providers/search`
Search providers by category, geography, SLA capability, and status.

### `POST /api/v1/providers`
Create provider profile and initial capabilities.

### `PATCH /api/v1/providers/{providerId}`
Update profile/capacity/certification metadata.

## 6.5 Billing

### `POST /api/v1/billing/subscriptions`
Provision or update tenant subscription mapped to Stripe objects.

### `GET /api/v1/billing/subscriptions/{subscriptionId}`
Retrieve internal + Stripe sync status.

### `POST /api/v1/billing/webhooks/stripe`
Stripe webhook ingestion endpoint.

Required event support:
- `checkout.session.completed`
- `customer.subscription.created`
- `customer.subscription.updated`
- `customer.subscription.deleted`
- `invoice.paid`
- `invoice.payment_failed`

## 6.6 Ads

### `POST /api/v1/ads/campaigns`
Create campaign with targeting, budget, and placement constraints.

### `GET /api/v1/ads/campaigns/{campaignId}`
Return campaign configuration and performance summary.

### `POST /api/v1/ads/events`
Ingest impression/click events (with anti-fraud metadata).

## 6.7 Reporting

### `GET /api/v1/reports/tenant-summary`
Returns KPI rollups:
- Lead funnel metrics
- Provider response SLA
- MRR/churn/payment recovery
- Ad CTR and ad-attributed revenue

## 7) Webhook and Callback Security

## 7.1 Stripe Webhooks

Requirements:
- Verify webhook signatures against Stripe signing secret
- Deduplicate by Stripe event ID
- Persist raw payload for audit and replay
- On processing failure: retry queue then dead-letter workflow

## 7.2 Partner Callbacks

Requirements:
- HMAC-signed payload validation
- Replay protection via nonce/timestamp windows
- Tenant and partner binding validation

---

## 8) Rate Limiting and Abuse Controls

Default policy (tenant-level + user-level):
- Auth endpoints: stricter per-IP controls
- Standard reads: burst + sustained bucket
- Mutations: lower burst thresholds
- Webhooks: separate ingress quota with queue buffering

Abuse protections:
- Bot/fraud scoring for ad click traffic
- Anomaly flags for repeated routing evaluate/dispatch patterns
- Dynamic throttling under incident mode

---

## 9) Observability and Audit Requirements

Every request must emit:
- Structured logs (JSON)
- `request_id`, `correlation_id`, tenant context, actor context
- Duration and status codes
- Redacted payload fields for PII/payment data

Audit events required for:
- Role/permission changes
- Billing plan or payment method changes
- Routing rule changes and overrides
- Manual dispatch interventions

---

## 10) Performance and SLO Targets

Aligned with enterprise spec:
- Standard read p95 latency: `< 300ms`
- Platform baseline availability: `99.9%`
- Horizontal scaling for stateless API tier

Endpoint-level guidance:
- Lead creation: p95 `< 500ms` (excluding async enrichment)
- Routing evaluate: p95 `< 700ms` for standard candidate pools
- Reporting summary: p95 `< 1.2s` with pre-aggregated rollups

## 11) OpenAPI 3.1 Structure Guidance

Recommended file layout:
- `openapi/openapi.yaml` (root)
- `openapi/components/schemas/*.yaml`
- `openapi/components/parameters/*.yaml`
- `openapi/paths/*.yaml`

Governance:
- CI validation for schema correctness and breaking changes
- Consumer contract tests for critical integrations
- Changelog entries required for endpoint/contract updates
